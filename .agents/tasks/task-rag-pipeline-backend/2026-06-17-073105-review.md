# RAG-powered contract drafting backend for KHS Material Ketenagalistrikan

A greenfield FastAPI backend that parses Indonesian contract templates from Excel, stores clauses in ChromaDB for retrieval, orchestrates an LLM (LMStudio) for draft generation, and exports results to DOCX. The architecture separates concerns cleanly: Excel parser, vector ingest, retriever, and a drafting service with template-fill fallback when the LLM is unavailable. The design anticipates a TipTap-based frontend editor.

**Watch for:** Silent exception swallowing in `generate_draft` hides LLM failures from callers (confirmed). CORS is fully open (`allow_origins=["*"]`) with credentials enabled — invalid per spec (confirmed). The HTML-to-DOCX export uses fragile regex parsing that will break on multi-line or nested HTML produced by the LLM (confirmed). The `ExportRequest.format` field is never validated against supported formats (confirmed).

## High-level view

The RAG layer uses ChromaDB's default embedding (all-MiniLM-L6-v2) rather than the configured `EMBEDDING_MODEL` setting, making that config field dead code.

The drafting orchestration fills variables via regex replacement first, then attempts LLM enhancement. If the LLM fails, the user silently gets the mechanical template-fill output without any indication that LLM enhancement was unavailable. The metadata response has no field distinguishing the two modes, so the frontend cannot display a degraded-mode indicator.

The export service converts HTML to DOCX using a line-by-line regex approach. This works for the self-produced HTML (one tag per line from `_direct_variable_fill`) but will silently drop or corrupt content when the LLM returns HTML with different whitespace patterns, nested tags, or multi-line elements.

Security posture is minimal: no authentication, no rate limiting, fully open CORS with credentials. The CORS configuration is actually broken — the spec forbids wildcard origins with credentials enabled, so browsers will reject credentialed cross-origin requests entirely.

<summary>Issues (7)</summary>

1. **Silent LLM failure in drafting** — `generate_draft` catches all exceptions from the LLM call and falls through to template-fill without logging or surfacing any signal. Add at minimum a `logging.warning` and a metadata flag (`llm_used: false`) so the caller knows what happened.
2. **CORS wildcard with credentials** — `allow_origins=["*"]` combined with `allow_credentials=True` is invalid per the CORS spec (browsers will reject it). Either restrict origins or drop credentials.
3. **Dead embedding config** — `settings.EMBEDDING_MODEL` is never referenced; `get_embedding_function()` always returns `DefaultEmbeddingFunction()`. Remove the config or wire it in.
4. **Regex HTML parser fragility** — `_parse_html_to_docx` assumes one HTML tag per line. LLM output will not conform to this. Use a proper HTML parser (BeautifulSoup is already a transitive dependency via ChromaDB).
5. **Unvalidated export format** — `ExportRequest.format` accepts any string but only DOCX is implemented. Add a `Literal["docx"]` constraint or validate at the route level.
6. **Unused import in dependencies** — `from typing import Generator` is imported but never used.
7. **No API route tests** — Tests cover schemas, text processing, and the RAG pipeline in isolation, but no tests exercise the FastAPI routes (TestClient). Error paths and HTTP status codes are untested.

## LLM fallback swallows failures silently

In `drafting_service.py` at the LLM call step:

```python
try:
    llm_html = await self.llm_service.generate_draft(...)
    if llm_html and len(llm_html) > 100:
        html_content = llm_html
except Exception:
    pass
```

Every exception type is caught and discarded — network failures, malformed responses, timeouts all vanish. The `len(llm_html) > 100` threshold is an arbitrary heuristic that could discard valid short responses (e.g., a single-clause contract). The metadata dict returned to the caller has no field distinguishing "LLM-enhanced" from "template-only" output, so the frontend cannot display a degraded-mode indicator or prompt retry.

## CORS configuration conflict (confirmed)

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    ...
)
```

The CORS spec forbids wildcard origins when credentials are allowed. Browsers will refuse to expose the response for any cross-origin request that sends cookies or auth headers. This is functionally broken, not merely permissive. The fix is either to enumerate allowed origins explicitly or to set `allow_credentials=False`.

## Regex HTML-to-DOCX brittleness (confirmed)

`ExportService._parse_html_to_docx` splits HTML by newlines and matches one tag per line:

```python
lines = html.split('\n')
for line in lines:
    h2_match = re.match(r'<h2[^>]*>(.*?)</h2>', line, ...)
```

When the LLM is used (the intended primary path), generated HTML will have arbitrary formatting: tags spanning multiple lines, nested inline elements, attributes containing quotes. The regex approach will silently skip unmatched lines, producing incomplete DOCX documents. Additionally, the `elements = re.findall(...)` result computed on line 97 is dead code — never used after assignment.

## Embedding model configuration is dead code

`settings.EMBEDDING_MODEL` defaults to `"nomic-embed-text-v1.5"` and is listed in `.env.example`, but `get_embedding_function()` ignores it:

```python
def get_embedding_function(model_name: Optional[str] = None):
    return embedding_functions.DefaultEmbeddingFunction()
```

The `model_name` parameter is accepted but never referenced. This creates a false impression that the embedding model is configurable. An operator changing the `.env` value will see no effect.

## Template routes duplicate clause-assembly logic

Both `list_templates` and `get_template` in `templates.py` contain identical loops for extracting variables and counting mandatory/optional clauses (~20 lines duplicated). As this logic evolves (e.g., adding variable types or descriptions from metadata), the two copies will diverge unless extracted into a shared helper.

## Multi-worker ChromaDB lock contention (possible)

The `RetrieverService` holds a `chromadb.PersistentClient` reference. Under gunicorn with multiple workers, each worker gets its own client pointing at the same SQLite-backed persistent directory. Ingestion currently happens only via the CLI script, so this is safe at present. If ingestion were ever triggered at runtime (e.g., an admin upload endpoint), concurrent writes would hit SQLite lock contention.

## Test coverage gaps

Not tested:
- FastAPI route handlers (no `TestClient` usage) — error responses, status codes, DOCX content-type header, Content-Disposition filename
- `DraftingService.generate_draft` integration (retriever + LLM together)
- `ExportService.html_to_docx` with realistic multi-element HTML
- LLM service retry logic and timeout behavior
- Streaming response behavior (`generate_stream`)

The Excel parser tests depend on the real template file at the configured path, making them environment-dependent.

## Export format validation gap (confirmed)

`ExportRequest` has a `format` field defaulting to `"docx"`, but nothing prevents a caller from sending `format: "pdf"`. The export route ignores this field entirely and always calls `html_to_docx`. Adding a `Literal["docx"]` type would reject invalid formats at the schema level rather than silently ignoring them.
